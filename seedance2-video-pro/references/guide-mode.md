# Guide Mode Reference

> Load when: 用户询问本 Skill 能做什么、怎么用、功能介绍、使用向导、支持哪些输入模式、生成视频前要准备什么，或问某个边界场景是否该触发本 Skill。
> Avoid: 用户已经明确进入真实生成、任务状态查询、API 参数排错或传统视频剪辑任务。
> Pairs with: `generation-workflow.md` 说明端到端流程；`visual-assets.md` 说明视觉资产路线；`cli.md` 说明命令入口；`completion.md` 说明结果复盘。

`guide mode` 是非生成视频场景下的强制说明模式。用户只要问功能介绍、使用向导、能力边界、生成前准备或“这个 Skill 怎么用”，就必须触发本模式。

用户有局部疑惑时，也走 guide mode 的按需功能说明，不要强行展开全套 8 项。比如用户只问“故事板有什么用”“角色卡什么时候做”“音频怎么管理”“全能参考是什么”“为什么不触发这个 Skill”，就只解释对应功能，但仍必须说明边界、需要的输入、确认门和下一步。

## Hard Rules

- 必须明确这是 guide mode，不会生成视频。
- 不调用 CLI。
- 不构造 payload。
- 不创建项目、资产、故事板或任务。
- 不声明已经生成、提交、等待或下载任何结果。
- 不要求用户先提供 API Key，除非用户问真实执行准备。
- 如果用户转为“现在生成”，再进入 `SKILL.md` 的完整生成执行协议。

## Required Guide Output

功能介绍不能只写一句话。默认至少覆盖以下 8 项；如果用户问得很窄，可以压缩，但不能漏掉“不会生成”和“下一步怎么开始”。

```text
这是使用向导，不会生成视频或调用 CLI。

1. 这个 Skill 解决什么问题
2. 它适合哪些视频任务
3. 它不适合哪些任务
4. 支持哪些输入模式
5. 复杂任务的标准流程
6. 需要用户准备哪些素材
7. 生成前会让用户确认什么
8. 下一步怎么开始
```

## On-Demand Feature Guide

当用户问具体功能、某个步骤、某类素材或某个边界时，按需解释，不要把完整功能介绍硬塞给用户。按需说明必须覆盖 5 项：

```text
这是使用向导，不会生成视频或调用 CLI。

1. 这个功能解决什么问题
2. 什么时候该用 / 什么时候不该用
3. 需要哪些输入或素材
4. 执行前会让用户确认什么
5. 下一步怎么继续
```

功能级答疑至少覆盖以下功能域：

| User asks about | Explain |
| --------------- | ------- |
| 输入模式 / 文生 / 图生 / 首尾帧 / 全能参考 | 6 个 Seedance 输入模式、素材角色、互斥边界和适用场景 |
| 剧本候选 / 自动拆分 / 多段视频 | 如何从创意估算时长、镜头数、角色/场景/道具需求，以及何时拆成多个 4-15 秒生成轮次 |
| 角色卡 | 角色卡用于锁人物身份；正式角色卡只用三视图或多视图模板；跨项目先检索既有角色卡，确认后复用或新生成 |
| 场景卡 | 场景卡用于锁空间、光线和道具；跨项目先检索既有场景卡，再按基本场景卡、正交四视图、多视角联合图按需选择 |
| 道具/产品卡 | 何时需要单独锁产品、道具、品牌物件和材质细节 |
| 故事板 / 分镜 / CUT 信息图 | 故事板母图是主运动和镜头参考；文字密集时可脚本拼版；生成前必须确认 |
| 声音 / 音效 / 说话人声 / 字幕 | 音乐、环境音、动作音效、对白/旁白/无说话人声、字幕和标题文字都要管理 |
| 结构化修改意见 / 确认复盘 | 生成前确认和结果复盘使用标准文本格式 + 结构化 JSON 字段；按 `workflow_stage`、`review_gate` 和 `requested_changes` 修改对应字段 |
| 生成前确认 | 模型、画幅、分辨率、时长、声音、素材编号、prompt、故事板和输出目录都要确认 |
| 结果复盘 / 迭代 | 汇报视频、输入、素材、证据和验收；不满意时只改 1-3 个点 |
| CLI / API / Key / doctor | 真实生成才需要 CLI 和 API Key；guide mode 只说明准备，不要求先配置 |
| 不触发边界 | 静态图、传统剪辑、其他 provider、Seedance 1.x/1.5 都说明替代路线 |

## Feature Guide Template

当用户有具体疑惑时，用这个短结构：

```text
这是使用向导，不会生成视频或调用 CLI。

关于 <功能名>：
- 作用：<它解决什么问题>
- 适合：<什么时候使用>
- 不适合：<什么时候不要用或要换路线>
- 需要准备：<素材/参数/确认信息>
- 生成前确认：<会让用户确认哪些输入>
- 下一步：<低摩擦继续方式>
```

当用户问角色卡或场景卡能不能跨项目复用时，必须明确回答：可以复用。流程是先用项目资产库检索既有角色/场景资产，展示候选并让用户确认；长期复用时用 `asset reuse` 登记到当前项目并保留 `reused_from`，快速试生成时可一次性引用旧素材路径，但要说明不会形成当前项目资产记录。

## Detailed Content Checklist

### 1. 解决什么问题

说明它不是普通 prompt 模板，而是视频导演工作流 + Seedance 2.0 API 执行代理：

- 把自然语言创意变成剧本、分镜、声音和参考素材计划。
- 根据素材选择文生视频、首帧、首尾帧、全能参考、编辑意图或延长。
- 通过 CLI 直连火山方舟 Seedance 2.0。
- 保存 prompt、payload、素材 manifest、generation log 和结果证据，便于复盘和迭代。

### 2. 适合任务

至少覆盖：

- 文生视频。
- 图生视频 / 首帧生成。
- 首尾帧生成。
- 多参考素材生成：角色图、场景图、故事板、参考视频、参考音频。
- 有声或无声视频。
- 短剧、广告、产品展示、角色动作、氛围片、社媒短视频。
- 视频生成模型上的编辑意图和延长。

### 3. 不适合任务

必须说清：

- 只要静态图时应走图片生成 Skill。
- 已有 MP4 只做剪辑、合并、转码、字幕压制时应走传统视频工具。
- 用户明确指定 Sora / Runway / Kling / Pika 等其他 provider 时，不默认使用本 Skill 执行。
- 当前 `edit` 不是 mask/inpaint/强制换皮。

### 4. 输入模式

至少列出 6 个模式：

- `text-to-video`：纯文本生成。
- `first-frame`：首帧 + prompt。
- `first-last`：首帧 + 尾帧 + prompt。
- `omni`：全能参考，支持参考图/视频/音频。
- `edit`：参考视频/图片 + 编辑意图 prompt。
- `extend`：参考视频 + 延长 prompt。

### 5. 复杂任务流程

简述标准流程，并引用 `generation-workflow.md`：

```text
创意 -> 剧本候选 -> 资产规划 -> 角色/场景/道具卡 -> 故事板 -> 生成前全输入确认 -> Seedance 生成 -> 结构化复盘
```

要说明每一步会提示已完成、下一步、需要素材、使用方法和是否确认继续。

### 6. 素材准备

按用户可能准备的内容说明：

- 文本创意或剧本。
- 角色参考图。
- 场景/背景参考图。
- 道具/产品图。
- 故事板或分镜。
- 参考视频。
- 参考音频。
- 希望的画幅、分辨率、时长、有声/无声和平台。

### 7. 生成前确认

说明真实生成前会确认：

- 选定剧本和段落。
- 预计时长、镜头数量、角色/场景/道具需求。
- 角色卡、场景卡和故事板提示词。
- 模型、画幅、分辨率、时长和声音。
- 所有参考素材编号和用途。
- 最终 `video-prompt.txt`。
- 是否真实调用 API、是否等待、是否下载。

### 8. 下一步

给一个低摩擦下一步，不要强迫用户立刻生成。例如：

```text
你可以给我一句创意，我先整理 2-3 个剧本方向和素材需求，不会直接生成；确认后再进入角色卡、场景卡和故事板阶段。
```

## Short Guide Template

当用户只问“这个 Skill 是什么 / 功能介绍”时，用这个结构：

```text
这是使用向导，不会生成视频或调用 CLI。

它用于把视频创意变成可执行的 Seedance 2.0 生成任务：先澄清意图，再规划剧本、角色/场景/道具、故事板、声音和参考素材，最后才通过 CLI 提交 API 并保存证据。

适合：文生视频、图生视频、首尾帧、全能参考、有声/无声视频、短剧/广告/产品展示、生成模型编辑意图和延长。
不适合：静态图、传统剪辑/转码/字幕压制、用户明确指定其他 provider 的任务。

标准流程：创意 -> 剧本候选 -> 资产规划 -> 角色/场景/道具卡 -> 故事板 -> 生成前全输入确认 -> 生成 -> 结果复盘。

真实生成前我会让你确认模型、画幅、分辨率、时长、声音、所有参考素材用途和最终 prompt。

下一步你可以给一句创意；我先只整理方向和素材需求，不会直接生成。
```

## Do Not

- 不要在 guide mode 里输出 CLI 命令作为“已经要执行”的下一步；只能说“真实生成时会用”。
- 不要把 guide mode 说成“可以顺便帮你生成一版”。
- 不要要求用户先给 Key 才能听介绍。
- 不要只列 capabilities，不说明流程、准备素材、确认门和边界。
