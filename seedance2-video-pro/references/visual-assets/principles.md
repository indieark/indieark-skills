# Visual Asset Principles

> Load when: 需要判断为什么复杂任务不能直接写视频 prompt，或需要决定设定图、故事板母图和 Seedance 之间的顺序。
> Avoid: 已明确只需要某一种图的提示词；直接读对应文件。
> Pairs with: `../visual-assets.md` 做入口路由；`setting-director.md` 生成项目设定；`seedance-handoff.md` 进入视频生成。

## Core Theory

复杂视频生成不是“写一条视频提示词”。稳定路径是先建立角色、场景和镜头资产，再让 Seedance 负责运动化。

三层链路：

```text
设定导演流
  -> 角色设定、场景设定、风格规则、段落逻辑
视觉压缩层
  -> 人物图、场景图、按需组织的故事板母图
Seedance 生成层
  -> 故事板主运动参考图 + 角色卡/场景背景卡细节参考 + 视频提示词
```

## Why Not Direct Prompt

直接把一句想法交给视频模型，容易出现：

- 角色漂移：脸、服装、年龄、比例在镜头间变化。
- 场景跳变：同一空间的门窗、光源、道具位置不连续。
- 镜头不连贯：模型不知道哪些画面是起点、推进、转折和收尾。
- 参考混淆：多张图一起输入时，模型自行判断主次，常把角色、场景、风格混在一起。

视觉资产的作用是提前固化这些不该由视频模型临场发明的内容。故事板母图是把创作意图压缩成可读参考的核心资产；九宫格、四栏、8 镜头或制作设定板只是它的具体版式，不要求每个场景严格按固定格数或固定栏目执行。

## Asset Layers

| Layer | Purpose | Failure Prevented |
| ----- | ------- | ----------------- |
| 人物设定图 | 锁定角色脸、服装、体态、配饰和识别点 | 换脸、换衣、年龄漂移 |
| 场景设定图 | 锁定空间结构、光源方向、材质和道具 | 换地点、光线跳变、空间穿帮 |
| 故事板母图 | 将角色、场景、镜头、调度、灯光、情绪、声音和色彩按需压缩到一张图，主控镜头顺序和运动 | 镜头无序、项目视觉不统一、模型参考混淆 |
| 版式实例 | 九宫格、四栏、8 镜头、制作设定板等 | 把版式误当硬模板 |
| 视频提示词 | 告诉 Seedance 如何读取故事板母图和各区域主次 | 模型把故事板当普通参考图 |

## Preferred Execution

单段短视频：

```text
设定导演流
-> 人物设定图 + 场景设定图
-> 故事板母图（可用九宫格或其他镜头网格）
-> Seedance 多参考输入：故事板图 + 角色卡 + 场景/背景卡 + 简短视频提示词
```

短剧、广告、系列项目：

```text
设定导演流
-> 人物设定图 + 场景设定图 + 风格规则
-> 故事板母图（可用四栏、制作设定板或自定义版式）
-> 用户确认
-> 拆单段时生成段落故事板，并把段落故事板、对应角色卡和场景/背景卡一起进 Seedance
```

只交一张故事板图是快速抽样或低复杂度路径。角色、背景或道具细节重要时，多图参考不是备选，而是必要输入：故事板图管镜头，角色卡管人物，场景/背景卡管空间和道具。每张图都必须在 prompt 中说明用途。

如果具体场景不适合九格或四栏，按场景改写故事板结构：可以把镜头减少到 4-6 个关键瞬间、扩展到更多节拍，或把信息区换成“角色/空间/调度/灯光/声音/色彩”等更贴合任务的组织方式。改写后仍要保存提示词，并在 `video-prompt.txt` 解释实际读图顺序和区域主次。

## Confirmation Gates

复杂任务不要越过这些确认点：

1. 设定确认：角色、场景、风格、画幅、时长。
2. 视觉资产确认：人物图、场景图、故事板母图是否符合意图。
3. 视频提示词确认：Seedance 实际会读哪张故事板母图、哪些角色卡和场景/背景卡会作为参考图、哪个区域决定运动、哪些参考只约束人物/场景/风格。
4. 生成确认：是否真实调用 API、是否等待和下载。

## Common Mistakes

- 人物图没定就做故事板，导致故事板里每格角色都不同。
- 场景空间没定就写运镜，导致镜头无法拍摄。
- 把九宫格、四栏或 8 镜头误当硬模板，而不是按需组织故事板母图。
- 故事板只是一张漂亮海报，没有时间顺序、区域主次或实际镜头逻辑。
- 只把故事板图交给 Seedance，没同时给角色卡和场景/背景卡，导致人物和空间细节从故事板小图里漂移。
- 视频 prompt 只写“参考故事板”，没有解释哪个区域决定运动、哪些区域只做约束。

## Self Check

- 是否让视频模型只负责运动化，而不是重新设计角色和场景？
- 是否把故事板图和角色卡、场景/背景卡一起作为视频生成参考输入？
- 是否保留了用户确认门？
- 是否能回溯每张图的用途和登记路径？
